# Lab 1.9 — Azure AI Foundry Workflows

You've built an agent, connected it to data, attacked it, defended it, and called it from code. This final lab introduces **Workflows** — Foundry's way of chaining multiple steps (models, agents, code, conditionals) into a single automated pipeline, all inside the portal.

This is a high-level walkthrough. The goal is to understand **what workflows are**, **when you'd use them**, and **where to go next** — not to build a production pipeline in 30 minutes.

---

## What Is a Workflow?

A workflow is a visual pipeline in Azure AI Foundry that connects steps together:

```
Input → Step 1 (LLM call) → Step 2 (code) → Step 3 (agent) → Output
```

Each step can be:
- A **model call** (prompt in, text out)
- A **code block** (Python — transform data, call APIs, apply logic)
- An **agent call** (your existing Foundry agent, with all its tools and guardrails)
- A **conditional branch** (if/else routing based on a previous step's output)

Think of it as the visual version of the multi-agent pipeline you wrote in Lab 1.8 — but built in the portal with a drag-and-drop interface.

### When Would You Use Workflows?

| Use a Workflow when... | Use Code (Lab 1.8) when... |
|---|---|
| Prototyping a multi-step pipeline quickly | You need full control over logic and error handling |
| Non-developers need to build or modify pipelines | Your team works in code and uses CI/CD |
| You want visual debugging (see each step's output) | You need custom integrations outside Azure |
| The pipeline is straightforward and linear | The pipeline has complex branching or retries |

Workflows and code are not mutually exclusive. Many teams prototype in workflows, then move to code for production.

---

## Create Your First Workflow

1. Go to [Azure AI Foundry](https://ai.azure.com) and open your project.
2. In the left sidebar, look for **Workflows** (or **Build → Workflows**).
3. Click **+ New workflow**.
4. Give it a name like `hello-workflow` or `studentXX-workflow`.

You'll see a visual canvas with an **Input** node and an **Output** node.

### Add a Model Step

1. Click **+ Add step** between the Input and Output nodes.
2. Select **LLM** (or **Model call**).
3. Choose your deployed model (e.g. `gpt-4o-mini`).
4. In the prompt, reference the workflow input:

   ```
   Summarize the following in one sentence: {{input}}
   ```

   The `{{input}}` placeholder pulls in whatever text is passed to the workflow.

5. Connect the model step's output to the Output node.

### Test It

1. Click **Run** or **Test**.
2. Enter some sample text as the input:
   ```
   Azure AI Foundry is a platform for building, deploying, and managing AI applications.
   It provides tools for model deployment, agent configuration, evaluations, and safety.
   ```
3. Watch the workflow execute step by step. You can click each node to see its input and output.

That's it — a one-step workflow that summarizes text. Simple, but it shows the pattern.

---

## Add a Second Step

Now chain a second step to see the power of workflows.

1. Add another **LLM** step after the first one.
2. In the prompt, reference the previous step's output:

   ```
   Translate the following to Spanish: {{steps.step1.output}}
   ```

   (The exact syntax for referencing previous steps may vary — check the UI for the correct variable name.)

3. Connect it to the Output node.
4. Run the workflow again.

Now your workflow: **summarizes → translates**. Two LLM calls, chained together, no code required.

---

## Connect Your Agent

The real value of workflows is connecting your existing agent — the one you built, hardened, and tested across Labs 1.2–1.7.

1. Add a step and select **Agent** (if available) or use a model call with your agent's configuration.
2. Point it to your agent (e.g. `007`).
3. The agent step uses everything you configured: system prompt, knowledge sources, MCP tools, content filters.

This means your workflow inherits all the security controls you've already built. The agent step is not a raw model call — it's your hardened agent.

---

## Security in Workflows

Everything you learned in this course applies to workflows:

- **Content filters** still apply to every model and agent step
- **Prompt shields** still block injection attempts at each step
- **System prompts** are preserved when you call an agent step
- **Data access controls** (MCP, AI Search) are enforced per-agent

New considerations for workflows:
- **Step-to-step data flow** — data passes between steps. If Step 1 produces PII, Step 2 sees it. Design your pipeline so sensitive data doesn't flow where it shouldn't.
- **Error handling** — if one step fails (content filter, timeout, bad input), what happens to the rest of the pipeline? Test failure scenarios.
- **Access control** — who can edit the workflow? Who can run it? Use Foundry's RBAC to restrict access.

## Fine-Tuning in Foundry

You've used RAG (Lab 1.4) to ground your agent in external data. There's another way to customize AI behavior: **fine-tuning**. It's worth understanding the difference, even if you won't fine-tune a model in this course.

### What Is Fine-Tuning?

Fine-tuning takes a base model (like `gpt-4o-mini`) and trains it further on **your own dataset** — examples of inputs and desired outputs. The result is a new, custom version of the model with adjusted weights that reflect your data.

**Fine-tuning applies to models, not agents.** You fine-tune a base model to create a custom model deployment. An agent could then *use* that fine-tuned model, but the fine-tuning happens at the model layer — it doesn't know about system prompts, tools, or knowledge sources.

### Fine-Tuning vs. RAG

| | RAG | Fine-Tuning |
|---|---|---|
| **What changes** | Nothing — the model stays the same, it just retrieves data at query time | The model's weights are modified with your training data |
| **Best for** | Factual answers from specific documents, data that changes often | Changing the model's style, tone, domain vocabulary, or reasoning patterns |
| **Data freshness** | Always current (searches at query time) | Frozen at training time — you'd need to re-train to update |
| **Setup effort** | Create a search index, connect it to your agent | Curate training data (input/output pairs), run a training job, deploy the new model |
| **Cost** | Search queries + inference | Training compute + inference (custom model deployments can cost more) |
| **Example** | "Answer questions about our product reviews" → RAG retrieves relevant reviews | "Always respond in our company's brand voice" → fine-tuning adjusts the model's style |

### When Would You Use Each?

- **Use RAG** when the answer lives in your data and you need the model to find and cite it. This is what you did in Lab 1.4 — the agent searches reviews and returns grounded answers.
- **Use fine-tuning** when you want the model to *behave differently* — adopt a specific tone, follow a particular reasoning style, or understand domain-specific terminology without being told every time.
- **Use both** when you need a model that speaks your domain language *and* retrieves current data. Fine-tune for behavior, RAG for facts.

### Data Storage Considerations

Fine-tuning requires you to upload training data to Foundry. Where that data is stored depends on your project's storage configuration (Microsoft-managed vs. bring-your-own, as covered in Lab 1.1). If your training data contains sensitive information — customer interactions, internal documents, proprietary examples — make sure your storage settings align with your organization's data governance requirements.

Fine-tuning is available in Foundry under **Fine-tuning** in the left sidebar. You won't use it in this course, but it's a natural next step once you've mastered RAG and agents.

## Monitoring and Governance

Two areas you'll encounter in production that go beyond what this course covers:

### Azure Monitor and Application Insights

In this course you tested your agent manually and ran evaluations. In production, you need **continuous monitoring** — tracking every agent interaction in real time:

- **Token usage and cost** — how much is your agent costing per day?
- **Latency** — how long are responses taking? Are MCP/search calls slowing things down?
- **Error rates** — how often are content filters blocking requests? Are tools failing?
- **Usage patterns** — what are users actually asking? Are there spikes or anomalies?

Azure AI Foundry integrates with **Application Insights** to give you dashboards, alerts, and query logs. You can set up alerts for things like "notify me if more than 5% of requests are blocked by content filters in an hour" — which could indicate an attack.

### AI Gateway

Foundry projects can enable **AI Gateway** — a free, centralized control plane that sits in front of your agents, tools, and models. AI Gateway provides:

- **Security** — centralized authentication and access control across all your AI resources
- **Telemetry** — unified logging and diagnostics for every agent call, tool invocation, and model request
- **Rate limiting** — throttle requests per user, per agent, or per model to prevent abuse and control costs
- **Observability** — trace requests end-to-end across agents, MCP tools, and model calls

Think of AI Gateway as a reverse proxy for your AI stack. Every request flows through it, giving you a single place to monitor, secure, and govern all AI traffic — regardless of whether the request comes from the playground, your code (Lab 1.8), or a workflow.

AI Gateway is enabled at the project level in Foundry. It's free to set up and is worth enabling even for development projects.

### Microsoft Purview

**Microsoft Purview** is Microsoft's data governance platform. When your AI agent handles sensitive data — customer PII, financial records, health information — Purview helps you:

- **Classify and label data** — automatically detect and tag sensitive information (PII, credentials, financial data) across your data sources
- **Track data lineage** — see where data flows: from your database, through the MCP server, into the agent, and out to users
- **Enforce policies** — set rules like "PII must not leave this data boundary" and have them enforced across services
- **Audit and compliance** — maintain a record of who accessed what data, when, and through which AI system

Purview integration is especially relevant for the MCP scenario from Lab 1.5. Your agent queries a database with customer PII — Purview can help ensure that data is classified, that access is logged, and that governance policies follow the data even as it moves through AI pipelines.

This is the enterprise governance layer that sits above everything you've built in this course.
