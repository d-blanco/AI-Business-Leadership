# AI Overview

AI is becoming a common tool in the enterprise, but the value it provides depends heavily on how problems are framed and where AI is allowed to operate. This lab introduces prompt engineering as a practical skill by showing how different levels of context and structure can dramatically change AI output across real workplace tasks.

You will compare multiple AI tools using the same prompts and data, observing how prompt quality, model behavior, and tool capabilities affect results, productivity, and risk. The goal is to understand not just how to use AI effectively, but how to use it responsibly within an enterprise security boundary.

## Key Terms
#### Gen AI
A type of AI that generates new content such as text, images, code, or data summaries rather than just analyzing existing data.

#### Prompt
The input you give an AI model that defines the task, context, and constraints. The quality of a prompt directly impacts the quality and safety of the output.

#### Foundation Models
Large, pre-trained models that serve as a base for many AI applications. They are trained on broad data and then adapted for specific tasks.

#### Word Tokenizer
The process an AI model uses to break text into smaller pieces (tokens) that it can understand and process.

#### Token
A chunk of text processed by a model, which can be a word, part of a word, or a symbol. Models reason and respond based on tokens, not sentences. Models are often priced based on the number of tokens they consume and produce (see [OpenAI Pricing](https://openai.com/api/pricing/) ).

#### Context Window
The maximum amount of text (tokens) a model can consider at once, including both your input and the model’s response.

#### Parameters
Internal values learned during training that determine how a model processes information and generates responses. More parameters generally mean more capability, but also higher cost.

#### Temperature
A setting that controls randomness in responses. Lower values produce more consistent and predictable outputs, while higher values allow more variation and creativity.

#### LLM (Large Language Model)
A type of foundation model focused on understanding and generating text-based content.

#### Multi-Modal
Models that can work with more than one type of input, such as text, images, audio, or video.

#### RAG (Retrieval-Augmented Generation)
A technique where an AI model retrieves information from external data sources before generating a response, improving accuracy and grounding.

#### Agent
An AI system that can take actions, such as modifying files, running commands, or interacting with applications, rather than only generating text.

## AI Provider Comparison

> Note: These tools and capabilities are subject to change. This comparison reflects the state of the platforms at the time of writing (early 2026).

### OpenAI
OpenAI popularized modern large language models and offers ChatGPT and API products. Model names and available features change frequently — confirm the current model/version on OpenAI’s website.

OpenAI’s handling of customer data (including retention and whether content is used to improve models) depends on the specific product, plan, and contractual terms. Enterprise agreements may provide options to restrict or opt out of training, so organizations should review OpenAI’s documentation and contracts before sending sensitive or proprietary data.



### Anthropic

Anthropic develops the Claude family of models with a strong emphasis on safety and enterprise features. Anthropic’s commercial terms and documentation describe how customer data is handled, and some enterprise plans include explicit commitments about data use and training.

Claude Code and related products provide agent-style developer workflows that can interact with codebases and files when configured. These capabilities can improve productivity but should be used with appropriate access controls and contractual assurances.

### Gemini

Google’s Gemini models are offered across Google products and developer platforms and support multi-modal inputs.

Data handling, retention, and use for model improvement vary across Google products (for example, consumer services vs. Google Cloud/Vertex AI) and are governed by product documentation and contractual terms. Organizations should consult Google’s documentation to understand integration and governance before processing sensitive data.

### GitHub

GitHub Copilot is an AI coding assistant integrated into development environments (for example, VS Code) that provides code suggestions and explanations.

GitHub’s documentation and enterprise settings describe telemetry, data handling, and options available to organizations. Historically, training data for code models included public code and other sources, and licensing/attribution considerations may apply to suggested code. Organizations should review Copilot documentation and contracts to confirm data handling and licensing/provenance implications.

### Microsoft

Microsoft 365 Copilot and related Microsoft AI offerings are positioned for enterprise use and emphasize tenant-level controls, compliance, and auditing.

Specific commitments about storage and training-use can vary by product and contract. Some enterprise offerings provide options that limit model training on customer content and keep processing within the organization’s tenant or approved cloud boundaries. Organizations should validate Microsoft’s documentation and their purchasing agreement to confirm data residency, training-use limitations, auditing, and retention guarantees.
### Azure AI Foundry

Azure AI Foundry is Microsoft's platform for building, deploying, and managing AI agents and applications. It's what you'll use throughout this course.

A key Foundry concept is **data storage options**. When you create a Foundry project, you choose where your data lives:

- **Microsoft-managed storage** — Foundry handles storage for you. Simpler to set up, but you have less control over where data resides and how it's managed.
- **Bring your own storage (BYO)** — You connect your own Azure Storage account, giving you full control over data residency, encryption keys, network rules, and retention policies.

This choice matters for security and compliance. Organizations with strict data governance requirements (HIPAA, FedRAMP, data sovereignty) typically use BYO storage so they control the encryption, access policies, and geographic location of their data. For prototyping and learning (like this course), Microsoft-managed storage is fine.
## Prompts

What you get back from AI depends on how well a prompt is structured. A good prompt considers how many possibilities an LLM has when answering. It aims to narrow those possibilities with domain expertise. Consider the examples below.

```
I need a website for uploading pdfs to cloud, and using AI to scan the PDFs and detect sensitive data. 
```

The prompt above is very open-ended; you may get some Python code that connects to Google Drive, OneDrive, Azure Blob Storage, or any cloud provider. Or, the model may ask you for more details. What libraries will it use?

> For best results, in your prompt ask the model if it has any questions before proceeding.

Consider how the prompt below is more detailed and requires domain expertise to conceive.

```
Use FastAPI to build an API with an endpoint to accept and upload a file. The endpoint should use Boto3 to upload the file to an S3 bucket. Read the bucket name and AWS access keys from environment variables. Before you proceed, do you have any questions?
```

It is always a good idea to have your prompts focused on small, achievable tasks. It is easier to understand and review smaller changes rather than large ones. Notice that the more detailed prompt above is a smaller, specific task that would be one step within a larger project. This is why domain knowledge is still required — you need to know *what* to ask for before AI can help you build it.

A good prompt is what makes the difference between someone who claims AI is not helping them solve their problem, and someone who masterfully uses AI to offload tasks.

- Customer impact was moderate, with some requests failing during the incident window
- Service was restored by rolling back the deployment to the previous stable version

Requirements:
- Use a neutral, factual tone suitable for leadership review
- Include sections for Summary, Impact, Root Cause, Detection, Resolution, and Preventive Actions

Before proceeding, ask any clarifying questions if needed.

## Conclusion

This lab demonstrated that effective use of AI in the enterprise is not about finding the “best” model, but about how problems are framed and where AI is allowed to operate. Through progressively refined prompts and real workplace tasks, you observed how small changes in structure and context can dramatically improve accuracy, usefulness, and reliability.

You also saw that AI capability alone is not enough. Tooling, agent access, and data boundaries play a critical role in determining whether AI can be used safely with real organizational data. As AI becomes more embedded in daily workflows, mastering prompt engineering alongside an understanding of enterprise security constraints will be essential for using these tools responsibly and effectively.
